## 1. 감사 범위 결정 (Codex 직접 수행)

- 전체 감사인지, 특정 시스템 감사인지 결정
- 비교할 문서와 코드 범위를 명시
- PROJECT-RULES.md의 문서 우선순위 확인

## 2. 불일치 수집 (Codex 직접 수행 또는 Claude Code Evaluator 위임)

대형 범위면 Claude Code에게 Evaluator 패킷으로 위임:
```
Read PROJECT-RULES.md and CLAUDE-HARNESS.md first. Act as Evaluator.
Audit the following scope for doc-code inconsistencies.
[scope, expected rules, failure conditions]
```

수집할 불일치 유형:
1. 문서 "완료"인데 실제 파일/경로/테스트/바인딩 없음
2. 문서 "미구현"인데 실제 구현 있음
3. 문서 용어/수치/규칙이 코드와 다름
4. research 문서의 파일 목록/역할/gaps가 실제와 다름
5. 새로 추가된 필드의 snapshot/restore 누락
6. 상위 문서와 하위 문서 충돌
7. temp plan에만 있고 상위 문서에 미반영된 결정

## 3. 분류 및 수정 계획 (Codex 직접 수행)

00-공통-보충규칙.md의 문제 등급 적용:
- A급 (즉시 수정): 빌드 에러, 테스트 실패, 명세 위반, 상태 경계 위반
- B급 (같은 작업에서 정리): 완료/미완 표기 오류, 바인딩 누락, research 불일치, 용어 혼용
- C급 (범위 내에서만): 표현 정리, 후속 메모 보강

큰 미구현 기능은 억지로 구현하지 말고 정확한 상태 문구로 남긴다:
`미개발` / `부분 구현` / `후속 구현 필요` / `실제 저장소 기준 재확인 필요` / `기획 판단 필요`
