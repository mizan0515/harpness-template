# 다음 작업 — 계획 및 실행 — Codex Orchestrator 절차

이 프롬프트는 Codex가 읽고 실행합니다. Codex가 다음 작업을 선택한 뒤,
Claude Code에 Generator/Evaluator로 위임하여 완료합니다.

---

## 자율 연속 실행 규칙

**이 절차는 한 번의 sprint 후 멈추지 않습니다.**
PASS → commit/push 후 즉시 1단계로 돌아가 다음 작업을 시작합니다.
사용자 재확인을 기다리지 마세요.

중지 조건:
- 사용자가 명시적으로 중단
- 설계 결정이 필요한 문서 충돌
- 위험한 git 작업이 필요한 경우
- 안전하지 않은 git 상태 (관련 없는 더티 파일로 인해 클린 커밋 불가)
- Codex 토큰/시간 한도 도달
- 동일 항목에서 3회 FAIL

Claude Code 호출 정책 (비용 관리 — AGENTS.md 비용 관리 정책 참조):
- 고정 접두사: `claude --model claude-opus-4-6 --effort low -p --add-dir . --permission-mode bypassPermissions "Read CLAUDE.md and CLAUDE-HARNESS.md first. ..."`
- **effort `low` 기본값** — `high`는 다중 파일 아키텍처 리팩토링 또는 원인 불명 디버깅에만 사용.
  - 단일 파일 편집, 데이터 생성, 리서치 업데이트, 테스트 추가 → 절대 `high` 사용 금지.
- **fresh session 기본값** — continued는 즉시 FAIL 후 1개 파일 이하 수정에만 사용.
- **packet당 1-3개 파일** 목표, 최대 5개. 초과 시 분할.
- **승인 대화상자 → 접두사 승인** ("유사한 명령을 앞으로 승인").
- **Evaluator 생략 가능**: 문서 전용 수정, 리서치 업데이트, 주석/로그 변경, 단일 오탈자 → Codex가 직접 검증.
- **20줄 이하 / 2개 파일 이하 명백한 편집**: Codex가 직접 수행 (Claude Code 비용 제로).
- 하나의 packet 완료 → 결과 검토 → 즉시 다음 packet 발행.

루프 흐름:
```
[1. 분석] → [2. 후보] → [3. 선택] → [4. 계약] → [5. Generator]
  → [6. 검토] → [7. Evaluator] → [8. 판정: PASS→commit→1 / FAIL→수정→5]
  → 반복 (사용자 대기 없음)
```

## 1. 현재 상태 분석 (Codex 직접)

- `git status`, `git log --oneline -10`
- 프로젝트 인덱스 문서 읽기
- 라이브 소스 파일 인벤토리 확인 (문서만 믿지 말 것)
- 구현 완료 항목과 문서화 항목 비교
- 최근 채팅 로그 확인 (있는 경우)
- 관련 `*-research.md` 파일 확인

## 2. 후보 수집 및 현실 점검 (Codex 직접)

- 현재 마일스톤에 대한 주요 갭 목록 작성
- 각 후보에 대해 검증: 실제 파일 존재, 런타임 경로 동작, 테스트 존재, 바인딩 존재
- 이미 해결된 항목 제외
- 과대한 항목 분할

## 3. 우선순위 결정 및 작업 하나 선택 (Codex 직접)

우선순위 순서:
1. 현재 마일스톤을 완료하는 연결 작업
2. 실제로 깨진 런타임 경로
3. 누락된 바인딩, 와이어링, 또는 설정
4. 테스트가 없는 핵심 프로덕션 코드 경로
5. 문서/리서치에서 잘못된 완료 상태
6. 콘텐츠 확장

작업 크기 제한:
- 소스 파일 1-3개 / 스키마 1개 / UI 화면 1개 / API 엔드포인트 1개 / 데이터 마이그레이션 1개

제외:
- 한 번의 sprint에서 완료하기에 너무 큰 작업
- 아직 문서화되지 않은 설계 결정이 필요한 작업
- 현재 마일스톤에 기여하지 않는 대규모 확장

## 4. Sprint 계약 작성 (Codex 직접)

AGENTS.md Sprint Contract Format에 따라:
- Sprint 이름, 목표, 사용자 대면 결과
- In Scope, Out of Scope
- 변경 가능성 있는 파일, 변경하면 안 되는 파일
- 완료 정의
- 테스트 시나리오, 실패 조건
- 리스크 / 가정
- 수행할 검증 확인 사항

## 5. Claude Code Generator Packet 발행

10-generator-packet.md 템플릿 사용.
Packet 시작 문구: `Read CLAUDE.md and CLAUDE-HARNESS.md first. Then execute the Generator Packet below.`

Packet에 포함해야 할 내용:
- 현재 상태 요약 (5-10줄)
- Sprint 목표, 필수 결과
- In/Out Scope
- 허용/금지 파일
- 제약 조건 (CLAUDE.md 가드레일 + 이 작업에 관련된 00-공통-보충규칙.md 함정)
- 도메인 특정 규칙/수치
- 구현 참고 사항
- 인계 전 필수 자체 점검
- 인계 형식

## 6. Generator 결과 검토 (Codex 직접)

- 변경된 파일이 scope 내인가?
- scope 외 변경이 정당한가?
- 보고된 리스크가 합리적인가?
- 리서치 업데이트, 검증 수행 완료?
- 가드레일 위반 없는가?

## 7. Claude Code Evaluator Packet 발행

11-evaluator-packet.md 템플릿 사용.
Packet 시작 문구: `Read CLAUDE.md and CLAUDE-HARNESS.md first. Then execute the Evaluator Packet below.`

Packet에 포함해야 할 내용:
- 평가 대상 Sprint
- 기대 동작
- 완료 정의 (4단계에서 가져옴)
- 필수 테스트 시나리오
- 명시적 실패 조건
- 04-도메인-체크리스트.md의 도메인 체크리스트 항목
- 검증 확인 사항
- 출력 형식: PASS/FAIL + 근거

## 8. 판정 및 후속 조치 (Codex 직접)

- PASS → commit/push → **즉시 1단계로 복귀** (사용자 대기 없음)
- FAIL → 수정 packet 작성 → 5단계로 복귀 (사용자 대기 없음)
- FAIL (설계 문제) → sprint 계약 수정 → 4단계로 복귀
- 동일 항목 3회 FAIL → 중지 후 사용자에게 보고

## 9. Sprint 메모 (Codex 직접, 간략히, 1단계 재시작 전)

- 완료된 작업 / 변경된 파일 / 검증 결과
- 남은 갭 (1-3개)
- commit/push 결과
- (이 메모는 내부용 — 사용자 응답을 기다리지 말고 → 1단계 진행)
